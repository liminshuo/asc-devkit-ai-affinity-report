P0 · 抓不到
机器发现层缺失 缺失 未入库 P0
测试问句
「asc-devkit 文档站有 llms.txt 吗？爬虫如何发现机器源文件？」
实测结果
GET https://gitcode.com/llms.txt → HTTP 200，但响应体为 SPA HTML 壳层（3,576 bytes），含 <script type="module"> 等前端 JS，非 LLM 协议要求的机器可读文本。
页面 <head> 无 <link rel="llms"> 或 <link rel="alternate" type="text/markdown"> 声明。
/sitemap.xml 返回空 <urlset>。
问题洞察
搜索引擎和 RAG 爬虫无法通过标准协议（llms.txt / sitemap）发现 asc-devkit 文档的机器可读版本。即使 robots.txt 已放行文档路径，也缺少指引爬虫优先抓取哪些 URL 的导航文件，导致入库覆盖率偏低、版本漂移不可控。
根因分析
  1. GitCode 平台未为用户仓库提供自动生成 llms.txt 的机制
  2. asc-devkit 项目本身未在仓库根目录维护 llms.txt 文件
  3. sitemap 路由被 SPA 路由拦截，返回空文档
Barnett FP
FP1 · 机器发现层失效：爬虫无法可靠地枚举文档 URL 集合
关联原则：→ #1 llms.txt / sitemap 部署
原始 Markdown URL 返回 SPA 壳层 待改进 抓取差 P0
测试问句
「这个文档有对应的 Markdown 源文件可以直接访问吗？」
GET https://gitcode.com/cann/asc-devkit/raw/branch/master/docs/quick_start.md
实测结果
/raw/branch/master/docs/quick_start.md → HTTP 200，但返回 SPA HTML（3,576 bytes），非 Markdown 文本。
/blob/master/docs/quick_start.md → SSR 渲染正常（249 KB HTML，含 50 <p> 标签和代码块）。
无 <link rel="alternate" type="text/markdown"> 头部声明。
问题洞察
RAG 管道若尝试通过 raw URL 获取纯 Markdown 源文件，将得到 HTML 壳层而非原始内容，导致解析失败或入库噪音。blob 页虽有 SSR 内容，但混入大量 Vue 组件 HTML，增加 chunk 切片难度。
根因分析
  1. GitCode SPA 路由拦截了 /raw/branch/ 路径，未在服务端区分 raw 请求与页面请求
  2. 缺少 Content-Type: text/markdown 的专用 raw 端点或链接声明
关联原则：→ #24 双轨交付
P1 · 找不准
版本信息完全缺失 缺失 语义未结构化 P1
测试问句
「我在看 gitcode.com/cann/asc-devkit/blob/master/docs/quick_start.md，这是哪个版本的文档？」
实测结果（联网大模型回答）
搜索引擎同时返回 blob/master/ 和 blob/9.1.0/ 两个版本的 quick_start.md，模型无法区分哪个是主推稳定版。JSON-LD 无 softwareVersion 字段。页面 <head> 仅显示分支名 "master"，无 SemVer。
问题洞察
RAG 无法区分 master 开发版和 9.1.0 稳定版文档，极易将两版本内容混合入库，导致用户收到版本不一致的回答（如新旧 API 用法并存）。
根因分析
  1. JSON-LD @type 为 SoftwareSourceCode，仅含 4 个字段，无 softwareVersion
  2. 页面固定位置（H1 下方）无版本 badge 展示
  3. 没有文档将 master 分支与 9.1.0 标签的关系显性化（如"master 对应下一版 ≥9.2"）
关联原则：→ #6 版本外显 · #8 元数据丰富化
无更新日期 · 鲜度不可机器识别 缺失 语义未结构化 P1
测试问句（DOM 审计）
检查所有采样页 <time datetime> 标签和 JSON-LD dateModified 字段。
实测结果
全部采样页面：<time> 标签数 = 0；JSON-LD 无 dateModified / datePublished；页面显示"22 minutes ago"等相对时间（仅视觉，非机器可读）。
问题洞察
RAG 系统无法判断文档鲜度，无法优先返回最近更新的内容；搜索引擎也无法使用 Last-Modified 来调控抓取频率，增加过时内容留存风险。
关联原则：→ #7 明确更新日期
JSON-LD 字段残缺 · 页面元数据贫乏 缺失 语义未结构化 P1
实测结果（DOM 审计）
所有页面 JSON-LD 均为固定模板：@type: SoftwareSourceCode · 仅含 name / codeRepository / programmingLanguage / author 四字段。
缺少：@type=TechArticle（或 HowTo）· headline · description · datePublished · dateModified · softwareVersion · inLanguage · keywords
问题洞察
搜索引擎和 RAG 管道无法通过结构化数据识别页面类型（教程/API 参考/概念文档），导致召回排序和 chunk 权重分配不准确。
关联原则：→ #8 元数据丰富化
废弃 API 无结构化状态标注 待改进 语义未结构化 P1

昇腾社区 vs GitCode asc-devkit

同一算子开发场景 · 平台对比 · 2026-07-07

① 一页结论 · 30 秒读懂

比的是什么？同样是 Ascend C 算子开发文档，昇腾社区是官网渲染站（hiascend.com），GitCode asc-devkit 是仓库 Markdown 文档。

谁整体更好？GitCode 更适合作为 RAG 内容源——Markdown 源更干净，步骤/代码/验证区块更完整；但两边在机器发现层和元数据上都严重缺失，AI 仍可能答错版本。

先改什么？两边都应先做 P0：#1 部署 llms.txt + #24 打通机器可读源（昇腾放行 source 路径，GitCode 修复 raw 路由）。

对比维度昇腾社区GitCode asc-devkit一句话
机器能不能发现文档 差 — llms.txt 404；source 被 robots 封 差 — llms.txt 是 SPA 壳；raw 路由坏了 都差，机制不同
内容能不能被 AI 读懂 差 — 图/Tab/热区大量丢失 较好 — Markdown 源，步骤/代码干净 GitCode 更好
版本/元数据是否可信 差 — 8.x 与 9.0 混用 差 — master 与 9.1.0 无 SemVer 都差

⑤ 各自先改什么 · 行动清单

P0 共同短板（都要补）

  • 页级版本元数据：JSON-LD / softwareVersion / dateModified，减少「最新版」口语
  • 装饰图 alt 规范化：导航 logo 可用空 alt；信息图必须描述性 alt + 图意

P0 昇腾应优先对齐 NVIDIA

  • 部署站级 / 产品级 llms.txt（参考 NV 的场景化链写法）
  • robots 放行 /doc_center/source/，sitemap 收录 source
  • 提供真实 Markdown/纯文本双轨（不要 SPA 壳）
  • 修 Tab 隐藏语义；补图意转写与热区文本清单；去掉代码行号入库

P1 NVIDIA 可继续加强

  • 抽样页补 TechArticle / HowTo JSON-LD
  • 页头固定展示 Toolkit 主推版本 badge（机器 + 人读）
  • 统一装饰图 vs 信息图的 alt 策略
  • 根 /sitemap.xml 可做跳转/索引，降低发现成本
